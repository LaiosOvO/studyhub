    - Add a `get_valkey` dependency function. Import `Valkey` from `valkey.asyncio`. Use `get_settings()` to read Valkey URL. Create client with `Valkey.from_url(settings.valkey_url)`. Handle connection failure gracefully (log warning, skip publish).
    - In `sync_experiment_status`, after the DB commit, publish a JSON message to `experiment:{run_id}` channel via `await valkey.publish(channel, json.dumps(progress_data))`. The progress_data dict should match the existing WebSocket payload shape: `{run_id, status, current_round, max_rounds, best_metric_name, best_metric_value, baseline_metric_value, rounds, gpu_metrics}`.
    - Wrap the publish in try/except (non-fatal — if Valkey is down, sync still works via REST).

    **4. Replace polling WebSocket with Valkey subscriber** in `experiment_progress`:
    - After accepting the WebSocket, send an initial snapshot from DB (same as current behavior for first message).
    - Then subscribe to `experiment:{run_id}` Valkey channel using `pubsub = valkey.pubsub(); await pubsub.subscribe(channel)`.
    - Listen for messages in a loop: `async for message in pubsub.listen()`. When type=="message", send message["data"] to WebSocket client.
    - On terminal states (completed/failed/cancelled) in the received message, break the loop and close.
    - Fall back to the existing 2s polling if Valkey connection fails (log warning, enter old polling loop).
    - Use `get_valkey` as an optional dependency (not injected, created inline since WebSocket endpoints don't support Depends well).

    **5. Add `queue_position` to ExperimentRunResponse** in schemas/experiment.py:
    - Add `queue_position: float = 0.0` to ExperimentRunResponse.

    NOTE: Do NOT use Redis library. Use `valkey.asyncio.Valkey` which is the project's standard (Valkey replaces Redis in this project).
    NOTE: Check if `get_settings().valkey_url` exists — if not, add `valkey_url: str = "valkey://localhost:6379"` to the Settings class in `app/config.py`.
